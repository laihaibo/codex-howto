# 任务规划

复杂任务先规划后执行，提高成功率和代码质量。

## 何时使用计划模式

- 任务涉及 3+ 个步骤
- 需要修改多个文件
- 有多种实现路径可选
- 需要权衡不同方案

## 计划模式使用

### 进入方式

询问 Codex 是否要进入计划模式：

```
> 这个任务比较复杂，先进入计划模式讨论方案
```

### 计划结构

```markdown
# 实现计划

## 目标
简要描述要达成什么

## 方案对比

### 方案 A：[名称]
- 优点：
- 缺点：
- 复杂度：

### 方案 B：[名称]
- 优点：
- 缺点：
- 复杂度：

## 推荐方案
选择方案 X，理由是...

## 实施步骤
1. [ ] 步骤 1：数据模型
2. [ ] 步骤 2：API 端点
3. [ ] 步骤 3：前端组件
4. [ ] 步骤 4：测试覆盖

## 风险和缓解
- 风险 1 → 缓解措施
```

## 多步骤执行策略

### 顺序执行

任务有依赖关系时依次执行：

```
步骤 1：创建数据库表
步骤 2：实现 Repository 层
步骤 3：实现 Service 层
步骤 4：实现 API 端点
步骤 5：实现前端组件
```

### 并行执行

无依赖的任务可并行：

```
并行任务：
- 任务 A：实现用户模块
- 任务 B：实现产品模块
- 任务 C：实现订单模块
```

### 增量验证

每个步骤完成后验证：

```bash
步骤 1 完成 → 运行 lint/typecheck
步骤 2 完成 → 运行单元测试
步骤 3 完成 → 手动验证
步骤 4 完成 → 集成测试通过
```
