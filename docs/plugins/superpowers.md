# Superpowers 插件

Superpowers 插件提供了一整套软件开发方法论和最佳实践技能，覆盖从需求分析到代码审查的完整开发周期。它是提升开发效率和代码质量的超级工具箱。

## 功能概述

- 🧠 创意工作前的头脑风暴
- 📝 多步骤任务计划编写
- ✅ 测试驱动开发（TDD）
- 🐛 系统化调试
- 👥 子代理并行开发
- 🔍 代码审查工作流
- 🏗️ Git 工作树隔离开发
- 📋 计划执行审查
- ✨ 完成前验证

## 包含的技能

### 技能列表

| 技能名称 | 描述 |
|---------|------|
| `brainstorming` | 创意工作前的需求探索和设计（在创建功能/组件前必须使用） |
| `writing-plans` | 多步骤任务的计划编写 |
| `executing-plans` | 在独立会话中执行计划并带有检查点 |
| `test-driven-development` | 测试驱动开发方法论 |
| `systematic-debugging` | 系统化调试（任何 Bug/测试失败/异常行为） |
| `dispatching-parallel-agents` | 派发独立子代理并行工作 |
| `subagent-driven-development` | 当前会话中的子代理驱动开发 |
| `finishing-a-development-branch` | 完成开发分支后的集成决策 |
| `using-git-worktrees` | Git 工作树隔离开发 |
| `requesting-code-review` | 请求代码审查 |
| `receiving-code-review` | 接收代码审查反馈 |
| `using-superpowers` | 发现和使用技能的基础技能 |
| `verification-before-completion` | 完成前验证（提交/PR 前运行验证命令） |

## 头脑风暴 (`brainstorming`)

**在任何创意工作之前必须使用** — 创建功能、构建组件、添加新功能或修改行为之前。

### 使用方法

在开始实现之前，使用此技能来探索用户需求、明确需求和设计方案。它会通过 relentless 访谈来打磨你的计划或设计。

### 触发时机

- 创建新功能
- 构建新组件
- 添加功能
- 修改现有行为

### 示例提示词

```
我想添加一个用户通知系统，请先帮我头脑风暴一下需求
```

```
帮我分析一下这个登录流程的设计方案
```

---

## 编写计划 (`writing-plans`)

为多步骤任务编写实施计划。

### 触发时机

- 有规范或多步骤任务需求
- 需要在开始编码前规划步骤
- 需要为子代理创建可执行的蓝图

### 示例提示词

```
帮我写一个实施计划来添加用户认证功能
```

```
为这个多步骤重构任务制定计划
```

---

## 测试驱动开发 (`test-driven-development`)

在任何功能实现或 Bug 修复前编写测试。

### 工作流程

1. 先编写失败的测试
2. 确认测试失败
3. 编写实现代码
4. 确认测试通过
5. 重构

### 触发时机

- 实现任何新功能
- 修复任何 Bug

### 示例提示词

```
用 TDD 方式实现这个用户注册功能
```

```
为这个 Bug 编写测试然后修复
```

---

## 系统化调试 (`systematic-debugging`)

在提出修复方案前，系统化地追踪 Bug 的根因。

### 触发时机

- 任何 Bug
- 测试失败
- 意外行为

### 工作流程

1. 重现问题
2. 收集证据
3. 形成假设
4. 验证假设
5. 实施修复

### 示例提示词

```
系统化地调试这个登录失败问题
```

```
这个组件渲染异常，帮我系统化排查
```

## 子代理模式

### 派发并行代理 (`dispatching-parallel-agents`)

当面对 2 个以上独立且没有共享状态或顺序依赖的任务时使用。

### 触发时机

- 多个独立任务可以并行处理
- 任务之间没有数据依赖
- 可以分配给不同的子代理

### 示例提示词

```
并行处理这三个独立的文件重构任务
```

```
同时修复这些不相关的 UI Bug
```

### 子代理驱动开发 (`subagent-driven-development`)

在当前会话中执行具有独立任务的实施计划。

### 触发时机

- 已有实施计划
- 计划中的任务相互独立
- 需要在当前会话中完成而非新会话

## 代码审查工作流

### 请求代码审查 (`requesting-code-review`)

完成任务、实现主要功能或合并前请求代码审查。

### 触发时机

- 完成任务后
- 实现主要功能后
- 合并前验证

### 接收审查反馈 (`receiving-code-review`)

收到代码审查反馈后，在实施建议前进行审查。

### 最佳实践

- 不要盲从审查意见
- 验证技术疑问
- 要求澄清不清楚的建议
- 必要时提出反驳

---

## Git 工作树 (`using-git-worktree`)

在开始需要隔离的特性工作前设置独立工作区。

### 触发时机

- 开始特性工作
- 执行实施计划前
- 需要独立工作空间

### 示例提示词

```
为这个特性分支创建一个 Git 工作树
```

---

## 完成开发分支 (`finishing-a-development-branch`)

实现完成、所有测试通过后，决定如何集成工作。

### 触发时机

- 实现完成
- 所有测试通过
- 需要决定合并、PR 或清理策略

### 示例提示词

```
我的特性分支完成了，帮我决定如何集成这个工作
```

---

## 完成前验证 (`verification-before-completion`)

在声称工作完成、修复或通过前，运行验证命令并确认输出。

### 核心原则

**证据先于断言** — 不要声称工作完成而没有实际的验证证据。

### 触发时机

- 声称 Bug 修复前
- 声称测试通过前
- 创建 PR 前
- 标记任务完成前

---

## 使用 Superpowers (`using-superpowers`)

基础技能，在**任何对话开始时**建立如何发现和使用技能的方法。

### 核心规则

- 在查找/使用技能前必须调用此技能
- 发现相关技能后使用 `Skill` 工具
- 用户的技能命名或任务描述匹配触发自动使用
